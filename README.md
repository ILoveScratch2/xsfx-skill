```
❯ 怎么写聊天软件？

● 先别急着上代码。聊天软件这东西，主要是先把约束想清楚，不然很容易写成一坨会动的史（）。

● 别一上来写 QQ。先做一个 能登录、能单聊、能存历史、能心跳保活、能断线补消息 的最小版。聊天软件本质上就是：长连接 + 
  消息路由 + 落库 + 重连恢复。这条链先打通，群聊、文件、加密都是后面往上堆。

   1. 先认约束：给自己玩，还是要真给几十上百人用？如果只是小圈子，别先搞微服务、消息队列、端到端加密。先验带动后验，需求没大到那个程度。
   2. 先搭骨架：前端随便，后端我会先用 Python + FastAPI + WebSocket。数据库本地先 SQLite，真上线再
  PostgreSQL。模块先切成 auth、ws、message、history。
   3. 协议先定死：先用 JSON 就行。比如
  `{"type":"send_msg","token":"...","to":"u123","msg_id":"m001","time":1710000000,"body":{"text":"hi"}}`

   服务端先支持 login、heartbeat、send_msg、ack、pull_history。
   4. 可靠性别后补：每条消息都要有 msg_id，服务端要回
  ack，客户端要发心跳包，断线重连后按最后一条消息补拉。没有这些，看起来能聊，其实一抖就丢。
   5. 边界要很硬：密码哈希、token
  校验、消息长度上限、发言频率限制、管理员权限。聊天软件最怕的不是功能少，是一上线就被刷爆。
   6. 开发顺序：本地单聊 → 历史消息 → 群聊 → 权限 → 文件/图片 → 加密。主路不稳就开旁路，WebSocket
  老挂的话，先用短轮询顶一下也不是不行。

  如果今天就开写，我会先写一个 单文件 WS 
  服务端：用户连上来，登录成功后能给指定用户发文本，服务端同时转发和落库。先把这个块跑起来。先别想着一步做到
  Telegram，不然很容易把自己写死 qwq
```
